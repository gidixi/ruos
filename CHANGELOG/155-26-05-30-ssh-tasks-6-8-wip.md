**Data:** 2026-05-30
**Status:** WIP ‚Äî kernel boots + accepts TCP, sunset `Runner::progress()`

## Cosa fatto

### Kernel ‚Äî multi-iface socket pool

`kernel/src/net/sockets.rs`:
- `POOL.alloc_tcp_eth()` aggiunge socket a `net.net_sockets`
  (Ethernet SocketSet) invece di `net.sockets` (loopback). Necessario
  per ricevere traffico from-outside.
- `SockEntry.ethernet` flag tracks which set.
- `POOL.is_ethernet(handle)` lookup.
- `tcp_get_mut`/`tcp_get` helpers dispatchano allo SocketSet corretto.
- `connect` ora usa `iface_net`/`iface_nic` context per socket Ethernet.
- `try_recv`/`try_send`/`close` non-blocking helpers (per SSH session loop).

### sunset_io.rs ‚Äî `run_session` (Tasks 6-8)

`pub async fn run_session(handle, host_signkey, authkeys)`:
- Box `inbuf` + `outbuf` 4 KiB ognuno
- `Runner::new_server(&mut inbuf, &mut outbuf)`
- Loop:
  1. Drain `runner.output_buf` ‚Üí `try_send`
  2. `try_recv` ‚Üí `runner.input(slice)`
  3. `runner.progress()` ‚Üí match `ServEvent`:
     - `Hostkeys(h)` ‚Üí `h.hostkeys(&[&SignKey::Ed25519(...)])`
     - `FirstAuth(a)` ‚Üí `enable_password_auth(false) + enable_pubkey_auth(true)`
     - `PasswordAuth(a)` ‚Üí `a.reject()`
     - `PubkeyAuth(a)` ‚Üí check pubkey contro authkeys, allow/reject
     - `OpenSession(s)` ‚Üí `accept()` (solo se auth_ok)
     - `SessionPty(p)` ‚Üí `succeed()`
     - `SessionEnv(e)` ‚Üí `succeed()`
     - `SessionShell(s)` ‚Üí alloc PTY pair via `try_claim` + spawn
       shell.wasm via `wasm::ssh_spawn::spawn_shell_on_pty` + `s.succeed()`
     - `SessionExec(e)` ‚Üí fail (phase-1 exec deferred ‚Äî no stdout tee)
     - `SessionSubsystem(s)` ‚Üí fail
     - `Defunct` ‚Üí break
  4. Channel ‚Üî PTY bridge:
     - SSH `read_channel(c, Normal, buf)` ‚Üí `pty::master_input_push(idx, b)`
     - `pty::master_output_try(idx)` ‚Üí `runner.write_channel(c, Normal, buf)`
  5. `Delay::ticks(1)` se non `progressed`

### PTY module additions

### Wasm ‚Äî shell on custom PTY

- Re-open `/dev/pts/<idx>` thrice

- `spawn_shell_on_pty(idx)` ‚Üí `enqueue_shell_pty`
`kernel/src/executor/mod.rs`:

### server.rs

- SESSION_CTX `Mutex<Option<SessionCtx>>` con `signing` + `authkeys`
- `spawn()` popola SESSION_CTX
- `serve_loop_pub()` attende ctx populated, accept loop reads ctx
- Per-session: clone signing key + authkeys, build `SignKey::Ed25519`,
  call `run_session`

## Bug aperto

`runner.progress()` hang sulla prima chiamata dopo TCP accept. Serial:
```
ssh client connected
ssh iter=1 outbuf=0 authok=false chan=false
ssh iter=1 pre-progress
[hang ‚Äî post-progress never logs]
```

Ipotesi:
- progress() panic con `panic=abort` ma kernel restante alive ‚Üí embassy
  task panic swallowed
- progress() entra loop interno (KEX init con force-soft crypto in
  debug build potrebbe essere multi-secondi ‚Üí 60s+ hang plausibile)
- smoltcp socket routing ‚Äî socket allocato in net_sockets ma altre
  helpers (recv/send) potrebbero non vedere il traffico

Next debug step:
- Build release profile (force-soft crypto in dev = lentissimo)
- Aggiungi panic-handler tracelog dedicato a SSH task
- Verifica MAC + IP packet flow via dmesg-style stats
