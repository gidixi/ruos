# 173 — WASI fd_readdir Task 3: host fn + handler

**Data:** 2026-05-31

## Cosa
Esportato `wasi_snapshot_preview1::fd_readdir` con ABI Preview 1
corretto. Completa la catena `opendir → fd_readdir` ([[171-26-05-31-fdentry-dir]],
[[172-26-05-31-path-open-directory]]).

- `kernel/src/wasm/suspend.rs`: nuovo `SuspendReason::FdReadDir { path,
  cookie, buf_ptr, buf_len, bufused_ptr }`.
- `kernel/src/wasm/host/fd.rs`: host fn `fd_readdir(fd, buf, buf_len,
  cookie, bufused_ptr)`. Risolve il fd: se non è `FdEntry::Dir` → `54`
  (ENOTDIR); altrimenti trappa con `FdReadDir`. Registrata in `link()`.
- `kernel/src/wasm/fiber.rs`: handler `FdReadDir`. Enumera via
  `vfs::readdir(&path).await`, costruisce la lista completa con `.`
  (idx 0) e `..` (idx 1) sintetici + le entry VFS, poi impacchetta
  record `__wasi_dirent_t` da 24 byte (d_next=i+1, d_ino=0, d_namlen,
  d_type da VfsKind) seguiti dai byte del nome. Salta le prime `cookie`
  entry (resume), riempie fino a `buf_len` ammettendo un'ultima entry
  troncata, scrive `bufused`. `readdir` error → bufused=0, ENOENT (44).

## Perché
`std::fs::read_dir` su `wasm32-wasip1` si abbassa a `fd_readdir`, che
non era esportato → qualsiasi crate che cammina directory (`walkdir`,
`glob`, `std::fs::read_dir`) falliva. Ora la catena standard
opendir/fdopendir/readdir funziona senza binding custom. `ruos.readdir`
(host fn 12-byte) resta intatto per `ls`/`find`/`du`/`grep`.

## Note ABI
- `d_type`: DIRECTORY=3, REGULAR_FILE=4, CHARACTER_DEVICE=2.
- `d_ino`=0 (std::fs non lo usa); nessun tipo SYMLINK (non abbiamo
  symlink).
- Cookie start = 0; `d_next = i+1` per ogni entry → resume con cookie=K
  salta le prime K. Entry finale parziale ammessa (wasi-libc ri-legge
  con buffer più grande).
