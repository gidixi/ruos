# 177 — WASM fuel metering: kill pure-compute loops, I/O-bound modules run forever

**Data:** 2026-05-31

## Cosa

Abilitato fuel metering in `wasmi` per proteggere il cooperative executor
da moduli WASM con loop infiniti senza host call.

### Modifiche a `kernel/src/wasm/fiber.rs`

- **`FUEL_PER_SLICE = 2_000_000_000`** — costante budget per fetta. Un loop
  puro (nessuna host call) brucia questo budget e viene terminato; un modulo
  I/O-bound rifornisce a ogni host call e gira indefinitamente.

- **`config.consume_fuel(true)`** in `Fiber::new` — abilita il contatore
  istruzioni in `wasmi::Config` prima di costruire l'`Engine`.

- **`store.set_fuel(FUEL_PER_SLICE)`** subito dopo `Store::new` in `Fiber::new`
  — semina il budget iniziale.

- **Rifornimento prima di ogni `resume`** nel run loop (`Fiber::run`): dopo che
  `dispatch(reason).await` ritorna e il check kill passa, prima di chiamare
  `state.resume(...)`:
  ```rust
  let _ = self.store.set_fuel(FUEL_PER_SLICE);
  ```
  Invariante: ogni volta che il controllo torna al wasm, il tank è pieno.
  Solo un loop senza host call può esaurirlo tra due `resume` consecutivi.

- **Kill su `ResumableCall::OutOfFuel`** — il branch già stub è stato rimpiazzato:
  ```rust
  ResumableCall::OutOfFuel(_) => {
      kprintln!("wasm: task killed (fuel exhausted)");
      return 137;
  }
  ```
  Restituisce 137 (convenzione SIGKILL) e lascia kernel + altri task illesi.

### API wasmi confermata

- `wasmi::TrapCode` — export pubblico diretto da `wasmi_core` in `wasmi::lib.rs:215`.
- `Error::as_trap_code(&self) -> Option<TrapCode>` — esiste, ma non è necessario
  qui: `ResumableCall::OutOfFuel` è già il variant corretto da matchare
  (non un `Err`). La variante porta `ResumableCallOutOfFuel` internamente.

## Perché

Il cooperative executor single-CPU non ha preemption. Un modulo WASM che non
chiama mai una host fn non restituisce mai il controllo → blocco totale del
kernel. Il fuel metering di wasmi impone un bound: ogni istruzione costa
carburante, le host call riforniscono. Solo un loop senza I/O può esaurire
il budget → trap `OutOfFuel` → kill pulito del solo task (exit 137), il kernel
continua a servire.
