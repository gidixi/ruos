# 194 — smp/pool.rs: kernel compute job pool

**Data:** 2026-06-01

## Cosa

Convertito `kernel/src/smp.rs` in modulo directory `kernel/src/smp/mod.rs`
(rinomina git, `crate::smp::bringup` invariato). Aggiunto sottomodulo
`kernel/src/smp/pool.rs` con:

- Array statico `SLOTS: [JobSlot; MAX_JOBS]` (MAX_JOBS = 64), ogni slot con
  campi atomici: `state` (u8), `work` (usize = fn pointer), `input_ptr`,
  `input_len`, `result` (u64), `ran_on` (u32 = cpu id).
- Stati slot: `EMPTY=0`, `QUEUED=1`, `RUNNING=2`, `DONE=3`.
- Coda `QUEUE: IrqMutex<VecDeque<usize>>` di slot id in ordine di submission.
- `JobFn = fn(&[u8]) -> u64` — pura CPU, nessuna cattura, nessun I/O.
- API pubblica: `submit(work, input: &'static [u8]) -> Option<usize>`,
  `take() -> Option<usize>`, `run_slot(id, cpu)`, `poll_done(id) -> Option<(u64, u32)>`.
- Transizioni via CAS SeqCst: `submit` EMPTY→QUEUED; `take` QUEUED→RUNNING;
  `run_slot` scrive result/ran_on + DONE; `poll_done` legge e libera (EMPTY).
- Trasmute fn pointer: `usize → *const () → JobFn` (round-trip sound su x86-64,
  fn pointer thin = un machine word).

## Perché

Task 1 SMP Fase 2: struttura dati pool + API per offload job pure-CPU verso AP
worker (task successivi). Il BSP fa `submit`; gli AP faranno `take`/`run_slot`;
il BSP raccoglierà risultati con `poll_done`.
