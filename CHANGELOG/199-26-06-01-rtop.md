# 199 — rtop (htop-like system monitor)

**Data:** 2026-06-01

## Cosa
Aggiunto `rtop`, monitor di sistema full-screen (htop di ruos): barre CPU%
per-core, barra memoria, uptime, conteggio task, tabella processi con
CPU%/MEM/TIME+/CMD ordinata per CPU% decrescente. Modalità `--once` batch
(testo, headless) usata anche come gate di test.

Kernel: accounting CPU basato su TSC che sfrutta il modello cooperativo (un solo
fiber attivo alla volta → CPU time = cicli TSC eseguiti, idle = cicli in hlt).
`ProcInfo` += `cpu_tsc`/`mem_bytes`, caricati per ogni burst wasmi in
`Fiber::run` (rdtsc attorno a `call_resumable`/`state.resume`; il gap
`dispatch().await` NON è contato). Nuovo `sched/cpustat.rs` con contatori
busy/idle per-core: BSP nel loop dell'executor (attorno a poll/hlt), AP nel
worker loop SMP (attorno a run_slot/hlt). Host fn `cpustat` (contatori per-core
+ tsc_per_ms) e `proc_stat` (proc_list esteso); `meminfo` riusato per la barra
memoria di sistema. `proc_list`/`ps` invariati.

App `user/rtop/` (std `wasm32-wasip1`): rendering con `ratatui` 0.29
(`default-features=false`) su un backend ANSI custom — crossterm/termion non
compilano su WASI (termios ioctl assenti). Raw mode via le host fn
`tcgetattr`/`tcsetattr` esistenti (guard RAII che ripristina termios + alt-screen
su uscita/drop; layout termios reale = 56 byte, c_lflag@12, da
`kernel/src/pty/termios.rs`). Parser dei blob LE coperti da unit test host
(`cargo test`, 4 verdi).

**Auto-refresh stile htop**: nuovo host fn `poll_stdin(buf, timeout_ticks)` +
`pty::slave_read_one_timeout` — legge un byte da stdin facendo race con un timer
direttamente sul PtyPair (NON via `vfs::read`/`with_fd_take`, che toglie la fd
dalla tabella e la lascerebbe orfana se il future venisse droppato sul timeout).
Il loop di rtop attende ≤1s un tasto: keystroke → reazione immediata (`q`),
timeout → ridisegno (refresh ~1 Hz), EOF → uscita. Verificato end-to-end via SSH
(`make run-rtop-test`): 4 frame in idle, `q` esce pulito, terminale ripristinato.

## Perché
North-star del pivot 2026: app utility reali sopra il runtime WASM. Un system
monitor mette alla prova accounting CPU senza preemption, host fn per i dati di
sistema, e una TUI su PTF (console locale + SSH). Conferma anche che `ratatui`
gira su `wasm32-wasip1`, sbloccando future app TUI.

## Limiti noti
- MEM per-processo = dimensione della linear memory wasm (pages×64KiB), non heap
  kernel reale. CPU% per-core: gli AP risultano ~idle salvo job paralleli attivi.
- CPU%/processo = 0 in una finestra di campionamento idle (corretto: niente cicli
  bruciati); sotto carico reale (es. uno spinloop) mostra valori non nulli.
