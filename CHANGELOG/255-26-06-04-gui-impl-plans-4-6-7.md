# 255 — Piani implementativi #4/#6/#7 (GUI runtime + ABI)

**Data:** 2026-06-04

## Cosa
Scritti i tre piani concreti che restano lato kernel per il desktop egui, ora
basati sulle API reali (framebuffer, mouse, PTY, ricetta AOT dello spike):
- #7 `wasmtime-runtime-router`: WtState + accessor memoria, WASI Preview 1 su
  `Linker<WtState>` (riuso servizi kernel), `run_cwasm`, blocking via epoch,
  router `.cwasm`→Wasmtime, pipeline Makefile `.cwasm`. (Foundation.)
- #4 `ruos-gfx-abi`: servizio `gfx` (geometria+blit+GUI_MODE), sospensione
  console, coda input + diversione tastiera, host fn `gfx_info/blit/poll_event`.
- #6 `ruos-proc-terminal`: host fn `pty_open/proc_spawn/proc_poll/winsize`,
  `spawn_on_pts` (tool wasmi su PTY, pattern ssh_spawn), exit tracking in proc.

Ordine: #7 fonda il linker, #4/#6 lo estendono. L'app GUI (#5) e l'emulatore
terminale sono PC-side (W:\Work\M\ruos-desktop); le ABI qui devono combaciare col
crate `abi` di quel repo.

## Perché
Gate Wasmtime superato (#3 GO); questi piani trasformano lo spike in runtime
usabile + host fn per la GUI. Implementazione successiva.
