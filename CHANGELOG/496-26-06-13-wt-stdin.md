# 496 — stdin interattivo sul runtime Wasmtime (.cwasm)

**Data:** 2026-06-13

## Cosa

I tool `.cwasm` (Wasmtime) ora possono leggere da stdin: prima `fd_read` su
fd 0 ritornava sempre EOF (i tool `.wasm` su wasmi avevano già stdin via
fiber/suspend; il path wt no).

- **`WtState.stdin_pts: Option<usize>`** (`wt/state.rs`): il pair PTY da cui
  leggere fd 0. `run_cwasm` lo imposta quando un PTY è bound (insieme a
  `stdout_pty`); ereditato dal main fiber dei moduli threaded.
- **`pty::slave_read_blocking(idx, out) -> usize`** (`pty/mod.rs`): lettura
  stdin bloccante per un consumer SYNC (un `.cwasm` su un AP, senza
  fiber/executor su cui sospendere). Drena il ring slave (cooked line
  discipline) in `out`, bloccando finché ≥1 byte è disponibile (semantica
  read(2)); `0` = EOF (pair shut down o app foreground `^C`/killed). Parca il
  core con `sti; hlt` tra i check (tick LAPIC dell'AP a 100 Hz → ~0% CPU,
  riga raccolta entro ≤10 ms). Come `poll_key`: gira SOLO off-BSP (il core 0
  pompa la line discipline) → ritorna EOF subito sul BSP. Stesso vincolo SMP
  del tui.
- **`fd_read` fd 0** (`wt/wasi.rs`): se `stdin_pts` è settato, legge una
  iovec via `slave_read_blocking` (una iovec per call, short read legale →
  libc rilegge); altrimenti EOF come prima.
- **`tools/wt-readline/`** (nuovo): tool `.cwasm` che fa eco di stdin
  (`STDIN_ECHO:<riga>`, esce su `quit`/EOF). Staged in `/bin`.
- **`tests/wt-stdin-test.sh`** + `make run-wt-stdin-test`: via SSH `-tt -smp 4`
  lancia `wtreadline`, manda due righe + `quit`, asserisce gli echi.

## Limiti

- Solo line-oriented cooked (il ring slave consegna righe complete a Enter) —
  raw-mode char-by-char passa dallo stesso ring ma il termios va messo in raw
  dall'app (le TUI usano già il path component `poll_key`/`set_raw`).
- BSP-only boots (1-2 core): stdin = EOF (il BSP pompa la line discipline).
  Interattivo richiede ≥3 core (un ComputeApp), come rtop/tui.
- Un `.cwasm` bloccato su stdin occupa il suo AP (hlt, ~0% CPU) come ogni
  tool sync bloccato su I/O; dentro un fiber threaded blocca quel core finché
  non arriva una riga (raro: i tool threaded raramente leggono stdin).

## Verifica

- `make run-wt-stdin-test` (deterministico, no SSH): boota
  `user-bin/wtstdin-init.sh` che lancia `wtreadline` sul PTY di boot. Asserisce
  `WT_READLINE_READY` (instantiate + run + fd_write) E l'ASSENZA di
  `WTSTDIN_INIT_DONE` — `wtreadline` si BLOCCA in `slave_read_blocking` su un
  core ComputeApp invece di uscire: prova che fd 0 raggiunge il read bloccante
  e NON ritorna EOF (lo stub vecchio). TEST_PASS_WT_STDIN.
- `make run-test`: TEST_PASS (nessuna regressione: i tool non interattivi
  vedono stdin EOF come prima).
- Echo round-trip (digita una riga → `STDIN_ECHO:`): verifica MANUALE
  (`make run`, poi `wtreadline` e digita) — l'input interattivo su harness
  headless è fragile.

## Issue separato scoperto (NON di questa modifica)

Provando lo stdin via `ssh -tt` ho trovato che un tool **core `.cwasm`**
(anche `wtecho`, che NON tocca stdin) lanciato da una shell SSH interattiva
**col desktop compositor attivo** si pianta sul suo core ComputeApp (mute
cores=N, output mai consegnato; #PF demand-paging nella regione wasm). NON
riproducibile headless (`make run-test`, smoke wtreadline) né con SSH exec
non-interattivo. Orthogonale allo stdin (lo dimostra `wtecho`). Da
investigare a parte: interazione exec `.cwasm` su AP ↔ PTY SSH ↔ routing
input compositor. rtop (path component `run_tui_component`) NON è affetto.
